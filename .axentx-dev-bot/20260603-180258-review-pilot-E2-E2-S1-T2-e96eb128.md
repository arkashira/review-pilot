export const prioritizationLogic = {
  criteria: [

### 2. Updated Feedback Model (`models/Feedback.js`)
```javascript
// models/Feedback.js
export class Feedback {
  constructor(data) {
    this.id = data.id;
    this.text = data.text;
    this.impact = data.impact || 0;
    this.urgency = data.urgency || 0;
    this.priority = data.priority || this.calculatePriority();
  }

  calculatePriority() {
    const { thresholds } = prioritizationLogic;
    if (this.impact >= thresholds.high.impact && this.urgency >= thresholds.high.urgency) {
      return 'high';
    } else if (this.impact >= thresholds.medium.impact || this.urgency >= thresholds.medium.urgency) {
      return 'medium';
    } else {
      return 'low';
    }
  }
}
```

## 1. Acceptance Criteria
1. **Ranking Accuracy** – Each feedback item displayed on the dashboard must have a `priorityScore` computed from the configured criteria and be sorted in descending order of this score.  
2. **Filter Functionality** – The UI must provide a filter control (e.g., dropdown or checkboxes) that allows the user to show only feedback items whose computed priority level is **High**, **Medium**, or **Low**. Selecting a filter updates the list instantly without a full page reload.  
3. **Configurable Logic** – An admin‑exposed JSON configuration (`priorityConfig.json`) can define the weight of each criterion (e.g., impact, urgency, user segment). Changing this file and reloading the dashboard must immediately affect the ranking and the derived priority level (High/Medium/Low).  
4. **Deterministic Mapping** – The mapping from `priorityScore` to priority level must be deterministic (e.g., ≥80 → High, 50‑79 → Medium, <50 → Low). The thresholds are also configurable via the same JSON file.  
5. **Performance** – Rendering the filtered list for up to 5 000 feedback items must complete within 200 ms on a typical production server (Node.js v20, 8 vCPU).  
6. **Accessibility** – Filter controls must be keyboard‑navigable and have appropriate ARIA labels.
